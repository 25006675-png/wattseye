Phased build plan

  Phase 1 — Synthetic data generator (1.5 hours)

  File: ML/sensing/synthetic_history.py
  - Generates 30 days × 1-second resolution
  - Realistic Malaysian patterns: morning kettle ~7 AM, fridge cycling 24/7, AC 8-11 PM weekdays, weekend shift
  - Includes 2 seeded anomalies (one empty-room AC event, one slightly long fridge cycle)
  - Saves to CSV + SQLite the backend can read
  - Unblocks: demo, routine engine, heatmap, bill trend chart, ALL the "look like a system that's been running" UI claims

  Phase 2 — K-Means routine phases (3 hours)

  File: ML/routine/kmeans_phases.py
  - Trains on the synthetic 30-day output
  - k=4 clusters → morning / work / evening / sleep
  - Output: kmeans_phases.joblib + a classify_hour(features) API
  - Existing routine_engine.py calls this, then fills in per-phase appliance statistics

  Phase 3 — Isolation Forest anomaly (3-4 hours)

  File: ML/anomaly/isolation_forest.py
  - Features per appliance event: (peak_W, duration_min, hour_of_day, dayofweek)
  - Trains on synthetic "normal" events
  - Validates against injected anomalies in the synthetic stream
  - Output: isolation_forest.joblib + anomaly_score(features) → float
  - Optionally retrain on UK-DALE later for the "trained on real public data" claim

  Phase 4 — Linear Regression fridge health (4 hours)

  File: ML/anomaly/appliance_health_regression.py
  - This one needs real data. Synthetic would just be us encoding our own assumptions, which judges will see through.
  - Download UK-DALE fridge submeter (small subset — house 1 fridge is ~50 MB compressed)
  - Features: (hour_of_day, day_of_week) → duty_cycle_fraction
  - Train sklearn LinearRegression
  - At inference: predict expected duty cycle for current hour, flag residuals ≥ 30%
  - Output: fridge_health.joblib + health_score(current_features, actual) → dict

  Phase 5 — Wire into insight orchestrator (2-3 hours)

  - Add the three new engine results to insight_orchestrator.py
  - Each engine returns the standard EngineResult shape (status, priority, reasons, metrics)
  - Smoke test end-to-end with a sample event

  Total: ~14 hours of work, doable in 3 days at relaxed pace or 1.5 days focused.

  Why this sequence is the right order

  1. Synthetic generator first is the unblocker — until it exists, the prototype HTML is faking everything and you can't
  show routine/history claims to anyone.
  2. K-Means before Isolation Forest because K-Means cleanly partitions the data, and IF can later use per-phase normality
  (anomaly definition depends on which daily phase you're in).
  3. Linear Regression last because it needs UK-DALE download which is the only "external dependency" — get the easy ones
  done first.

  Where the demo numbers come from

  After Phase 1, the prototype HTML can read from the synthetic SQLite and every "RM 82 month to date" / "14 days observed"
   / "Empty-room AC 71 min" claim becomes a real query result, not a hardcoded string. That's a big credibility upgrade.

Excellent questions — these are real architecture concerns, not edge cases. Let me answer each clearly.

  1. Does synthetic data depend on the NILM model?

  No, fully independent. The synthetic generator produces ground truth — the actual per-appliance power traces. NILM is
  what tries to recover that breakdown from the total-power signal. They're separate concerns:

  Synthetic generator produces (the truth):
    timestamp, total_W, kettle_W, fridge_W, ac_W, occupancy
                  ↓                  ↓                ↓
          feeds NILM input    used as training    used to verify
          (total_W only)       labels for NILM    NILM's accuracy

  So switching NILM models later changes nothing about the synthetic data — only the model that consumes it. The same
  synthetic dataset works for ELECTRIcity, vanilla CNN, BERT4NILM, or anything else.

  2. Does retraining NILM force the whole stack to restart?

  Only if you couple them badly. With the right architecture, no.

  There are two ways to wire the downstream models (Routine, K-Means, IF, LR):

  | Pattern | What it does | Verdict |
  |---|---|---|
  | A. Downstream trained on NILM output | K-Means / IF / LR ingest whatever NILM predicts, so a NILM swap invalidates their training | Fragile — avoid |
  | B. Downstream trained on per-appliance ground truth or rolling history | Downstream models care about patterns over time, not which specific NILM produced them. Swap NILM, downstream catches up within weeks. | Recommended |

  We want B. The downstream models should be trained on either:
  - Synthetic ground truth (during prototyping — what we'll do)
  - Smart-plug measured values (in deployment, if any are installed)
  - The Pi's rolling 30-day history of NILM output (in deployment, with the understanding that this implicitly tracks
  whatever NILM is producing)

  If you swap NILM later, the downstream models keep working on the new stream. They'll converge to the new appliance
  estimate distribution within a few retraining cycles.

  3. How does retraining work in production?

  Per-model schedule, all local to the Pi:

  | Model | Retrain cadence | Why |
  |---|---|---|
  | NILM (ELECTRIcity) | Quarterly or yearly | Heavy training, mostly stable per-appliance signatures. Fine-tunes on the home's auto-labeled AC data (plan 03 §23 Use 3) |
  | K-Means routines | Monthly | Routines drift seasonally — school holidays, weather, work-from-home patterns |
  | Isolation Forest | Monthly | What counts as "normal" shifts as the household changes habits |
  | Linear Regression health | Quarterly + on confirmed repair events | Baselines should be stable; you don't want to "learn" a degrading fridge as normal |

  The architecture that makes this work: shadow training

  ACTIVE model serves predictions today
         ↓
  Pi accumulates 30 days of operational data
         ↓
  Background job trains SHADOW model on last 60 days
         ↓
  SHADOW evaluated against held-out 7 days
         ↓
  If SHADOW score > ACTIVE score → promote
  Otherwise keep ACTIVE, log decision

  Each model file is versioned: kmeans_v0.joblib (factory-shipped), kmeans_v1.joblib (first per-home retrain),
  kmeans_v2.joblib, etc. Old versions kept for one cycle in case of regression.

  Cold-start vs warm operation

  For a brand-new home install:

  | Time | What's running | What user sees |
  |---|---|---|
  | Day 1 | Factory-default models (shipped with synthetic + UK-DALE training) | Dashboard works, but "Routine still learning" badge |
  | Week 4 | First per-home K-Means + IF retrain triggers | Routine alerts become household-specific |
  | Month 3 | Linear Regression fridge baseline established | Health drift detection active |
  | Month 6 | NILM fine-tuning starts (uses auto-labeled AC data from dedicated clamp) | NILM accuracy improves on this home's specific appliances |

  This is the same cold-start pattern Nest, Sense, and Emporia use. The product is designed to be good on day 1 and better
  by month 6.

  What the Pi needs for this

  - Rolling SQLite DB of last 90 days of (timestamp, total_w, ac_w, per_appliance_estimates, occupancy) — about 5 GB max at
   1 Hz resolution
  - Cron / systemd timer that fires monthly retrain
  - Versioned model directory — models/active/, models/shadow/, models/archive/
  - Retraining log — what was promoted when, against which metric
  - Failsafe: if retraining crashes, ACTIVE keeps serving — never break running predictions

  This is ~200 lines of orchestration code total. No cloud roundtrip, no exotic MLOps stack.

  Net answer

  - Synthetic data is independent of NILM. Train it once, reuse forever.
  - A NILM swap doesn't restart the stack if downstream models are wired to consume output streams, not bound to specific
  NILM weights.
  - Production retraining runs locally on the Pi, monthly for the lightweight models and rarely for NILM. Each model is
  versioned and shadow-trained, so retraining never breaks live predictions.
